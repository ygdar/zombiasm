# ToyASM — Учебный ассемблер на языке C

## 1. Система команд

Минималистичная, но мощная архитектура — **ToyASM**. 16-битные слова, 8 регистров общего назначения, фиксированный формат инструкций.

### Регистры

`R0–R7` — регистры общего назначения. `R6` — Stack Pointer (SP), `R7` — Program Counter (PC).

Флаги: `Z` (zero), `N` (negative), `C` (carry).

### Таблица команд

| Опкод | Мнемоника | Формат | Описание |
|-------|-----------|--------|----------|
| 0x00 | `NOP` | — | Нет операции |
| 0x01 | `MOV Rd, Rs` | reg←reg | Копирование регистра |
| 0x02 | `LDI Rd, #imm` | reg←imm8 | Загрузка константы |
| 0x03 | `LD Rd, [Rs]` | reg←mem | Чтение из памяти |
| 0x04 | `ST [Rd], Rs` | mem←reg | Запись в память |
| 0x05 | `ADD Rd, Rs` | Rd = Rd+Rs | Сложение |
| 0x06 | `SUB Rd, Rs` | Rd = Rd−Rs | Вычитание |
| 0x07 | `AND Rd, Rs` | побитовое И | Маскирование |
| 0x08 | `OR Rd, Rs` | побитовое ИЛИ | Установка битов |
| 0x09 | `XOR Rd, Rs` | побитовое XOR | Инверсия битов |
| 0x0A | `SHL Rd` | Rd <<= 1 | Сдвиг влево |
| 0x0B | `SHR Rd` | Rd >>= 1 | Сдвиг вправо |
| 0x0C | `CMP Rd, Rs` | флаги по Rd−Rs | Сравнение |
| 0x0D | `JMP addr` | PC = addr | Безусловный переход |
| 0x0E | `JZ addr` | если Z — PC=addr | Переход если ноль |
| 0x0F | `JN addr` | если N — PC=addr | Переход если < 0 |
| 0x10 | `JC addr` | если C — PC=addr | Переход если carry |
| 0x11 | `CALL addr` | push PC; PC=addr | Вызов подпрограммы |
| 0x12 | `RET` | pop PC | Возврат из подпрограммы |
| 0x13 | `PUSH Rs` | [--SP] = Rs | Сохранение в стеке |
| 0x14 | `POP Rd` | Rd = [SP++] | Восстановление из стека |
| 0x15 | `INT #n` | программное прерывание | Системный вызов |
| 0x16 | `IRET` | восстановить контекст | Возврат из прерывания |
| 0x17 | `HLT` | стоп | Остановка процессора |

Этого набора достаточно для: алгоритмов сортировки, рекурсии, реализации стека вызовов, **кооперативной многозадачности через `INT` + `IRET`** (сохраняем R0–R5 + PC + флаги в TCB — Task Control Block, загружаем контекст следующей задачи).

## 2. Структура компилятора (ассемблера)

Компилятор построен по классической **двухпроходной схеме**. Ниже описаны все модули и этапы.

---

### Этап 1 — Лексер (tokenizer)

Разбивает поток символов на токены: `MNEMONIC`, `REG`, `LABEL`, `IMM`, `COMMA`, `NEWLINE`.

**Совет по реализации:** читай один символ за раз. Реализуй две вспомогательные функции: `peek()` — посмотреть текущий символ без продвижения, и `advance()` — взять символ и сдвинуть указатель.

```c
typedef enum {
    TOK_MNEMONIC, TOK_REG, TOK_LABEL, TOK_IMM,
    TOK_COMMA, TOK_NEWLINE, TOK_EOF
} TokenType;

typedef struct {
    TokenType type;
    char      text[64];
    int       line;
} Token;
```

---

### Этап 2 — Парсер строк

Каждая строка исходника преобразуется в структуру `Instruction`.

**Совет по реализации:** проверяй корректность числа операндов прямо здесь, до любых проходов.

```c
typedef struct {
    uint8_t  opcode;
    uint8_t  operands[2];   // номера регистров или immediate
    char     label_ref[64]; // ссылка на метку (если есть)
    int      line_num;
} Instruction;
```

---

### Этап 3 — Проход 1: сбор меток

Перебирает все инструкции, считает адреса (Location Counter — LC) и заполняет **таблицу символов**: `имя_метки → адрес`.

**Совет по реализации:** для таблицы символов подойдёт либо простой отсортированный массив с `bsearch()` из `stdlib.h`, либо хеш-таблица (реализуй простейшую на открытой адресации).

---

### Этап 4 — Таблица символов

Хранит соответствие `char* name → uint16_t addr`.

```c
typedef struct {
    char     name[64];
    uint16_t address;
} Symbol;
```

---

### Этап 5 — Проход 2: кодогенерация

Транслирует список инструкций в машинный код. Для каждой ссылки на метку ищет адрес в таблице символов.

**Совет по реализации:** функция `encode(op, rd, rs, imm)` возвращает `uint16_t`, который сразу пишется в файл через `fwrite()`.

---

### Этап 6 — Обработчик ошибок

Перехватывает ошибки на всех этапах:

- неизвестная мнемоника
- неверный операнд (например, `R8` при 3 битах на регистр)
- неопределённая метка

**Совет по реализации:**

```c
fprintf(stderr, "Ошибка строка %d: %s\n", line_num, message);
exit(1);
```

---

### Этап 7 — Кодировщик инструкций

Упаковывает поля в 16-битное слово с помощью побитовых операций.

**Формат слова:**

```
[15..10] opcode  (6 бит)
[ 9.. 7] Rd      (3 бита)
[ 6.. 4] Rs      (3 бита)
[ 3.. 0] imm4    (4 бита, для коротких immediate)
```

Для инструкций с `imm8` поля Rd/Rs сужаются или imm занимает биты 7..0.

**Пример кодирования:**

```c
uint16_t encode(uint8_t op, uint8_t rd, uint8_t rs, uint8_t imm) {
    return ((uint16_t)op  << 10)
         | ((uint16_t)rd  <<  7)
         | ((uint16_t)rs  <<  4)
         | (imm & 0x0F);
}
```

---

### Этап 8 — Генератор листинга (опционально)

Создаёт текстовый файл для отладки. Каждая строка: адрес | hex-код | исходная строка.

```
0x0000  0802    LDI R0, #2
0x0002  0841    LDI R1, #1
0x0004  1401    ADD R0, R1
0x0006  5C00    JMP 0x0000
```

---

### Этап 9 — Выходной файл (.bin)

Последовательность `uint16_t` в big-endian порядке.

**Совет по реализации:** добавь минимальный заголовок — магическое число и размер программы в словах, чтобы эмулятор мог проверить корректность файла.

```c
// Запись одного слова
uint16_t be = htons(word);  // big-endian
fwrite(&be, 2, 1, output_fp);
```

## 3. Ключевые решения при реализации

### Двухпроходная схема

Первый проход нужен для разрешения *forward references*: когда `JMP loop_end` встречается до определения метки `loop_end`. Без первого прохода пришлось бы делать backpatching — возвращаться и дописывать адреса в уже записанный бинарник. Это сложнее и ненадёжнее.

### Фиксированный формат инструкции

16 бит на слово, 6-битный opcode (достаточно для 64 команд), два 3-битных поля регистров, 8-битное immediate. Это упрощает декодирование в эмуляторе до одной строки маскирования.

### Кооперативная многозадачность

Реализуется через `INT #0` (yield). Обработчик прерывания в эмуляторе:

1. Сохраняет R0–R7 + флаги в TCB (Task Control Block) текущей задачи
2. Выбирает следующую задачу по round-robin
3. Восстанавливает её контекст
4. Выполняет `IRET`

Всё это можно написать прямо на ToyASM — отличный учебный проект.

```asm
; Пример yield — добровольная передача управления
yield:
    PUSH R0         ; сохраняем рабочие регистры
    PUSH R1
    INT  #0         ; передаём управление планировщику
    POP  R1         ; восстанавливаем после возврата
    POP  R0
    RET
```

## 4. Следующий шаг — эмулятор

После компилятора пиши эмулятор. Основная структура:

```c
typedef struct {
    uint16_t regs[8];       // R0–R7
    uint16_t mem[65536];    // 128 КБ памяти
    uint8_t  flags;         // Z | N | C
} CPU;
```

Главный цикл: **fetch → decode → execute**.

```c
void run(CPU *cpu) {
    while (1) {
        uint16_t word = cpu->mem[cpu->regs[7]++];  // fetch + PC++
        uint8_t op = (word >> 10) & 0x3F;           // decode opcode
        uint8_t rd = (word >>  7) & 0x07;
        uint8_t rs = (word >>  4) & 0x07;
        uint8_t im =  word        & 0xFF;
        // execute: switch(op) { ... }
    }
}
```
